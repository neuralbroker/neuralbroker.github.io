BlitzKode

Coding model · Qwen2.5-Coder-3B · SFT · GRPO · DPO · LoRA · GGUF · llama.cpp

BlitzKode started as a way to understand the full path behind a coding assistant: data, supervised fine-tuning, RL-style alignment, preference optimization, conversion, inference, and release. I wanted to see how useful a compact coding model could be for focused coding tasks, without treating it like a magic replacement for larger models.

What I built

I fine-tuned Qwen2.5-Coder-3B using SFT, GRPO, DPO, and LoRA with PyTorch and Hugging Face, then converted the result to GGUF for CPU-only inference with llama.cpp. LoRA, gradient checkpointing, and bfloat16 mixed precision cut training VRAM from roughly 24GB to 8GB.

Why it matters to me

This project connects directly to my interest in AI developer tools and coding agents. A coding assistant is only useful when it fits into a workflow, responds quickly enough, and can be evaluated against real tasks. BlitzKode is published as neuralbroker/blitzkode on Hugging Face with 510+ downloads and reproducible training configs.

What I learned

  • Fine-tuning is only one step; data quality, alignment, and evaluation matter just as much.
  • Memory-efficient training choices like LoRA, gradient checkpointing, and mixed precision can make larger experiments practical on limited hardware.
  • Packaging and inference details, especially GGUF and CPU-only deployment, change how usable a model feels.
