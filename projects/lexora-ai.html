Lexora AI

Per-user RAG Q&A. Sub-2s retrieval over 100k-token documents with isolated FAISS indexes.

code notes architecture

Problem

When many users share one RAG process, user A's question can retrieve chunks from user B's documents. Metadata filtering at query time is the wrong default for multi-tenant services.

Architecture

Upload → Chunk → Embed → FAISS (/user_id/)
  → Retriever (document_ids filter in search, not after)
  → Chat service (history as real turns, not flattened string)
  → SSE stream or single response

Tradeoffs

  • Why FAISS per user? Hard isolation. Retrieval service scoped to user_id cannot reach another user's index.
  • Why not Pinecone? Side project scale. FAISS is free, local, and sufficient for prototype-to-small-production.
  • Why Redis for JWT? jti denylist on logout + single-use refresh tokens. Cache keys include user_id + query hash for safe hits.
  • Why Celery opt-in? Same code path for inline (dev) and background (prod) document processing.

Lessons

Passing chat history as real user/assistant turns (not a flattened string) improved multi-turn accuracy more than any other single tweak. FAISS rebuilds on document delete — fine for low churn, needs in-place delete for production scale.
