Engineering

How I think about building systems — not a skill matrix, but principles, tools, and trajectory.

Principles

  • Software should be simple. Complexity is a loan you pay interest on.
  • Measure before optimizing. Guessing at bottlenecks wastes weeks.
  • AI augments humans. Automate the boring parts; keep humans in the loop for judgment.
  • Understand before abstracting. Read the code, then build the abstraction.
  • Documentation is engineering. If it's not written down, it doesn't ship.
  • The boring infrastructure is the work. Auth, caching, monitoring, and deployment matter more than the model.
  • Multi-tenant defaults must be secure. The unsafe path is always shorter — don't ship it first.
  • Evaluation is a separate project from training. Smoke tests are not benchmarks.
