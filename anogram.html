Anogram: Offline P2P Messaging over Bluetooth LE

Published: February 2026 · GitHub

Anogram is an Android messaging app that works without internet, accounts, or servers. Two people in the same room can open the app, discover each other, and start chatting — all over Bluetooth Low Energy.

It's built with Kotlin, Jetpack Compose, and Clean Architecture. Here's how the core idea works.

The problem

Most messaging apps (WhatsApp, Telegram, Signal) need internet. That's fine 99% of the time, but there are situations where no connectivity exists — remote areas, disaster zones, secure facilities, or just saving data. Bluetooth LE lets devices talk directly, but the traditional BLE model (one central, one peripheral) makes multi-device messaging awkward.

Core logic: BLE mesh

Anogram implements a mesh-like topology over BLE GATT. Each device acts as both a server (advertising a GATT service with a "message" characteristic) and a client (scanning for nearby peers and connecting to read their messages).

The flow works like this:

  1. Discovery — Each device advertises a custom BLE service. Nearby devices scan and find each other.
  2. Connection — A device connects as a GATT client to a discovered peer's GATT server.
  3. Message exchange — Messages are written to the peer's "message" characteristic. The peer's server notifies its client of new data.
  4. Local storage — All messages are persisted in a Room database on each device.

Architecture

Anogram follows Clean Architecture + MVVM with three layers:

  • Domain — Pure Kotlin models (Chat, Message) and repository interfaces. No Android dependencies.
  • Data — BLE implementation using Android's BluetoothLeScanner and BluetoothGatt APIs, plus Room DAOs for persistence.
  • Presentation — Jetpack Compose screens with Material 3, ViewModels using Kotlin Coroutines and Flow.

Dependency injection is handled by Hilt, making the BLE and database modules swappable for testing.

Why BLE instead of Wi-Fi Direct or NFC?

  • BLE — Low power, works on virtually all modern Android devices, no pairing UI needed for basic data exchange
  • Wi-Fi Direct — Faster for large data but more complex setup and higher battery drain
  • NFC — Too short range (4 cm) for convenient messaging

BLE is the sweet spot for text-based local messaging.

What I learned

Building Anogram meant learning the nuances of Android's BLE stack: service/characteristic UUID management, MTU negotiation for larger messages, connection intervals, and handling the many edge cases of device discovery (devices going out of range, advertisements being missed, etc.). It's a practical example of how Clean Architecture makes sense even in a hardware-near Android project — the domain models stay clean while the BLE details live in the data layer.
